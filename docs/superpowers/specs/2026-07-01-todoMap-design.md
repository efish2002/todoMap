# todoMap — 设计文档

> 日期:2026-07-01
> 状态:草案,待用户审核

## 1. 概述

todoMap 是一个本地桌面应用,用于记录"项目工作网络":节点是人,边是"我委托给 ta 的 todo"或"ta 委托给我的 todo"。同项目的人/边用同一种颜色。用户以"我"为中心看到自己的协作网。

MVP 阶段是单人版(只服务一个真实用户),数据完全本地(SQLite)。但**数据模型和导出格式**从第一天就设计成可迁移到多用户共享,后续扩展不会丢数据。

## 2. 目标用户

- 一个具体的、有协作关系的人(开发者 / 产品 / 设计师 / 团队负责人)。
- 痛点:在多个项目里同时和不同人协作,记不清自己派给谁什么事、谁欠我什么事、谁被卡住了。
- 关键体验:打开应用,一眼看到自己的协作网络,不需要在多个看板/IM 之间跳。

## 3. 范围

### 3.1 在范围内 (MVP)

- 本地桌面应用(Windows / macOS / Linux),数据存本地 SQLite。
- 单用户:有且仅有一个"我"节点,首次启动时由用户输入名字并上传头像。
- 三类实体:**人**、**项目**、**委托(todo)**。
- 关系图(节点-箭头图),"我"在中心,力导向布局,边带箭头。
- 边/节点边框颜色 = 所属项目颜色。
- 右侧边栏:点击图上任意节点后,显示该节点相关的全部 todo(我派给 ta 的 + ta 派给我的)。
- 委托的完整字段:标题、描述、状态、截止日、优先级、标签、评论。
- 项目管理:增删改、选颜色、归档;有专门入口,也可在新建委托时顺手建。
- 人的去重:录入 todo 时填名字,系统按"名字 + (可选)头像"自动去重,重名时弹确认。
- 搜索:按人 / 项目 / todo 标题搜索。
- 导出 JSON(全量数据)、从 JSON 导入(覆盖或合并)。

### 3.2 不在范围内 (MVP 后续考虑)

- 多用户共享 / 登录 / 权限。
- 同步到云端(本设计保证数据可导出,可作为未来同步的种子)。
- 提醒 / 通知 / 邮件。
- 时间轴 / 甘特 / 看板视图。
- 评论 @ 提及。
- 移动端。
- 主题切换(深色为主,以后再加浅色)。

## 4. 技术栈

| 层次 | 选型 | 理由 |
|------|------|------|
| 桌面壳 | Tauri 2 | 体积小、启动快、Rust 写 SQLite 稳。 |
| 前端 | React + TypeScript + Vite | 主流、生态好,React Flow 直接可用。 |
| 关系图 | React Flow | 节点拖拽 / 连线 / 缩放 / 力导向布局开箱即用。 |
| 数据 | SQLite,通过 Tauri Rust 端访问 | 前端只调 Tauri 命令,不直接连库。 |
| 样式 | 简单 CSS / CSS Modules | 不引入 UI 库,避免冗余。 |

## 5. 架构

```
┌─────────────────────────────────────┐
│  WebView (React + React Flow)       │  ← 渲染层
│  - 图、列表、表单                    │
│  - 通过 invoke() 调 Tauri 命令       │
└────────────────┬────────────────────┘
                 │ IPC
┌────────────────▼────────────────────┐
│  Rust 端 (Tauri commands)            │  ← 业务层
│  - CRUD:人、项目、委托                │
│  - 去重、查询、导出                   │
└────────────────┬────────────────────┘
                 │ SQL
┌────────────────▼────────────────────┐
│  SQLite                             │  ← 存储层
│  - people / projects / todos         │
│  - todo_links (委托方向)             │
└─────────────────────────────────────┘
```

## 6. 数据模型

```sql
-- 一个人
CREATE TABLE people (
  id            INTEGER PRIMARY KEY,
  name          TEXT NOT NULL,
  avatar_path   TEXT,           -- 本地文件路径(可选)
  is_me         INTEGER NOT NULL DEFAULT 0,  -- 1 表示是"我"
  created_at    TEXT NOT NULL
);

-- 一个项目(决定颜色)
CREATE TABLE projects (
  id            INTEGER PRIMARY KEY,
  name          TEXT NOT NULL,
  color         TEXT NOT NULL,  -- 十六进制 #RRGGBB
  archived      INTEGER NOT NULL DEFAULT 0,
  created_at    TEXT NOT NULL
);

-- 一条委托(我派出去 / 别人派给我)
CREATE TABLE todos (
  id              INTEGER PRIMARY KEY,
  title           TEXT NOT NULL,
  description     TEXT,
  status          TEXT NOT NULL DEFAULT 'pending',  -- pending/in_progress/done/blocked
  priority        INTEGER NOT NULL DEFAULT 0,        -- 0=无 1=P1 2=P2 3=P3
  due_date        TEXT,                              -- ISO YYYY-MM-DD
  tags            TEXT,                              -- 逗号分隔
  project_id      INTEGER NOT NULL REFERENCES projects(id),
  from_person_id  INTEGER NOT NULL REFERENCES people(id),  -- 委托方
  to_person_id    INTEGER NOT NULL REFERENCES people(id),  -- 接收方
  created_at      TEXT NOT NULL,
  updated_at      TEXT NOT NULL
);

-- 委托的评论
CREATE TABLE comments (
  id          INTEGER PRIMARY KEY,
  todo_id     INTEGER NOT NULL REFERENCES todos(id),
  author_id   INTEGER NOT NULL REFERENCES people(id),
  body        TEXT NOT NULL,
  created_at  TEXT NOT NULL
);
```

### 6.1 关键约束

- `people` 表里**有且仅有 1 条 `is_me = 1` 的记录**,由应用启动时检查保证。
- `todos.from_person_id != todos.to_person_id`(不能自己委托给自己)。
- `todos` 中至少有一个人必须是 `is_me = 1`,即每条委托都与"我"直接相关(单人版 MVP 限制)。
- 项目颜色唯一性不在数据库层强约束(允许重色,UI 上提示)。

### 6.2 人的去重规则

录入委托时,用户在"派给谁"输入框填名字:
1. 已有同名人的:**直接复用**,不弹窗(同名就是同一个人,简单优先)。
2. 同名但已有不同头像的:**弹确认框**"你输入的'小王'与已存在的'小王'(头像不同)是否是同一个人?",用户选"是"则合并,选"否"则创建新的人(允许显示名后加空格或序号区分)。

未来扩多用户时,去重逻辑会升级为"按 phone/email/user_id 等稳定标识",但当前阶段保持简单。

## 7. 界面

### 7.1 主界面布局

```
┌──────────────────────────────────────────────────────────────┐
│ todoMap  [项目(3)] [+新建项目]    搜索…    [+新增委托]  ⚙   │  ← 顶部
├──────────────────────────────────────────┬───────────────────┤
│                                          │ ┌───────────────┐ │
│                                          │ │ [头像] 我     │ │
│            关系图区域                    │ │ 3项目 · 7委托 │ │
│            (力导向 / "我"在中心)         │ ├───────────────┤ │
│                                          │ │ 我派出去的·3  │ │
│                                          │ │ ■ 写登录页API │ │
│                                          │ │ ■ 画原型      │ │
│                                          │ │ ■ 整理接口文档 │ │
│                                          │ │               │ │
│                                          │ │ 别人给我的·2  │ │
│                                          │ │ ■ 写联调脚本  │ │
│                                          │ │ ■ 审核设计稿  │ │
│                                          │ └───────────────┘ │
├──────────────────────────────────────────┴───────────────────┤
│ ● 3项目 ● 5人 ● 7委托 · 2进行中 · 1逾期     导出 JSON · 备份│  ← 底部
└──────────────────────────────────────────────────────────────┘
```

### 7.2 关系图规则

- 节点:圆形,带头像(或首字母),名字在头像下。
- 节点边框颜色 = 该人至少参与 1 个项目时,选最近编辑的项目颜色(若有多个则用优先级最高那个;MVP 简化为"最近编辑"1 个)。
- 边:带箭头,箭头从 `from` 指向 `to`(委托方 → 接收方)。
- 边颜色 = 该 todo 所属项目颜色。
- 布局:力导向,但**"我"节点固定在中心**,其他人围绕它排。
- 边和节点都支持点击 / 右键菜单。
- 鼠标悬停:显示 todo 标题 tooltip。

### 7.3 侧边栏规则

- 始终显示"我"的卡片(顶部)。
- 当选中某个节点 N:显示节点 N 的卡片,下方是该节点相关的 todo(我派给 N 的 + N 派给我的),分两组。
- 选中"我"时,显示"我派出去的"和"别人派给我的"两组(MVP 默认视图)。
- 选中空白处:仍显示"我"卡片。

### 7.4 顶部按钮

- **项目(N)**:点击展开项目列表(弹窗或下拉),每行可改颜色、改名、归档、删除。
- **+ 新建项目**:快速新建。
- **搜索框**:模糊匹配,搜索人/项目/todo 标题,回车跳到对应节点/打开 todo 详情。
- **+ 新增委托**:打开新增委托弹窗。
- **⚙(齿轮)**:导出 JSON、导入 JSON、关于。

### 7.5 新增委托弹窗

字段(从上到下):
1. 标题(必填)
2. 描述(可选,多行)
3. 委托方(下拉选人,默认是"我",也支持输入新名字)
4. 接收方(下拉选人,默认无,必填)
5. 所属项目(下拉选项目,默认无,必填;没有时可点"+ 新建项目"在该弹窗内快速建)
6. 状态(默认 pending)
7. 截止日(可选,日期选择)
8. 优先级(无 / P1 / P2 / P3)
9. 标签(逗号分隔)
10. 保存 / 取消

提交时校验:from != to,from 或 to 之一必须是"我"(MVP 限制)。

### 7.6 委托详情

点击图中的边,或侧栏的 todo 卡片,都会弹出该 todo 的详情:含完整字段展示、评论区(可增删自己写的评论,按时间倒序)。

## 8. 关键交互

1. **首次启动**:弹"你叫什么名字"+ 上传头像(可跳过用首字母)。保存后写入 `people(is_me=1)`。
2. **新增委托**:见 7.5。提交后图上自动多一条带箭头的边,颜色 = 项目颜色。
3. **录入时输"小王"**:复用现有"小王";若已有"小王"且头像不同则弹确认(见 6.2)。
4. **点击图中节点**:侧栏刷新为该节点相关 todo(我派给 ta 的 + ta 派给我的)。
5. **点击边**:弹委托详情(7.6)。
6. **右键节点**:编辑名字 / 上传头像 / 删除(若该人已无任何委托,允许删除;否则提示"先清理委托")。
7. **右键边**:编辑 todo / 删除。
8. **搜索**:回车跳到目标;若多人/多 todo 命中,弹下拉选。
9. **导出 JSON**:全表 dump 为一个 JSON 文件,带 schema 版本号(便于以后升级)。
10. **导入 JSON**:选覆盖(清空旧数据)或合并(按名字去重)。

## 9. 错误与边界

- 数据库损坏:启动时做完整性检查,损坏则弹"恢复模式"(从最近一次导出导入)。
- 重名的人:见 6.2。
- 项目被删除但还有 todo 引用:不允许硬删,提示"先迁移 todo 到其他项目或归档项目"。
- 项目归档:已归档项目不出现在默认下拉中,但可手动展开"已归档"看到。
- 图节点数过多(>50):自动开启"折叠到项目"模式,同项目的人合并显示为簇(MVP 不实现,作为未来优化)。

## 10. 性能与限制

- MVP 假设单用户数据量 < 1000 条 todo,< 200 人,SQLite 足够。
- 不做服务端,所有查询在前端发起,Rust 端用 prepared statement。
- 关系图渲染 < 200 节点时 React Flow 流畅,> 200 节点需要做"按项目折叠",MVP 暂不做。

## 11. 实施分阶段

1. **阶段 1 — 骨架**:Tauri 项目初始化 + SQLite 建表 + Rust CRUD 命令 + 启动时检查/创建"我"。
2. **阶段 2 — 关系图**:React 集成 React Flow,实现"我"在中心 + 边渲染 + 项目颜色。
3. **阶段 3 — 委托录入**:新增/编辑/删除委托弹窗 + 边的增删联动。
4. **阶段 4 — 侧栏**:选中节点刷新 todo 列表,显示详情弹窗。
5. **阶段 5 — 项目 / 搜索 / 导出**:项目入口、搜索、JSON 导入导出。
6. **阶段 6 — 打磨**:右键菜单、tooltip、空状态、错误提示、首次启动引导。

## 12. 开放问题 / 后续

- 多用户共享的数据模型升级:要不要引入"用户账号"概念?目前用 `is_me` 标记单人,扩展时要重构。
- 节点折叠 / 大图性能:超过 200 节点时的策略。
- 时间轴视图:是另一个独立模式,还是图上的"播放"功能。
- 数据备份策略:是只靠用户手动导出,还是应用内做自动本地快照?
